## 3. セクション5: 法務・コンプライアンス（有償提供の要件情報）

> 以下は一般的な制度・ガイドライン・公開事例の収集であり、法的助言ではありません。**規約・免責・保険・データ設計の最終確認は弁護士に依頼してください。**

### (1) 利用規約／免責事項に盛り込むべき条項（一般例）

- **正確性の非保証**：判定結果は告示・通知・各厚生局公表情報に基づく参考情報であり、正確性・最新性・完全性を保証しない。厚労省・各厚生局も「届出時に最新情報を確認」「チェックリスト提出は届出ではない」と明記しており、ツールの位置づけも同様に限定するのが整合的。
- **最終判断の帰属**：施設基準適合の最終判断・受理は地方厚生（支）局が行い、当社は届出の受理・算定可否を保証しない。
- **責任制限**：誤判定・改定追従遅延による逸失利益・返戻・自主返還等への責任を、契約上の範囲（例：直近一定期間の利用料相当を上限）に制限。BtoBのため一定の合理性はあるが、故意・重過失は制限不可が原則。
- **第三者リンクの免責**：厚生局PDFのURL変更・リンク切れに関する免責。
- **改定追従の努力義務化**：データ更新は合理的努力で行うが即時反映は保証しない。

### (2) 医療広告ガイドライン／景品表示法

- 医療広告ガイドラインは**医療機関の広告**を規律するもので、SaaS事業者自身のサービス広告は直接の対象ではない。ただし顧客（歯科医院）が導入をHP等に掲載する場合、医療機関側で適用を受け得るため、誤認を招く文言（「必ず受理される」等）の提供は避ける。
- 景品表示法は原則 BtoB 取引には適用されない（保護対象は一般消費者）。ただし優良誤認（「業界No.1」「100%正確」等の根拠なき表示）・有利誤認は避け、客観的根拠を保持。

### (3) 個人情報保護法（入力データを保存する場合）

- 医院名・医療機関コード・スタッフ研修歴等を保存する場合、利用目的の特定・通知公表、安全管理措置、委託先（Supabase等のクラウド）の監督義務が生じる。
- **設計境界の明確化が有効**：本アプリは「医院の体制情報のみを扱い、患者個人データ（要配慮個人情報）は扱わない」設計とすることで、リスクを大きく下げられる。現状の判定は決定木でステートレスに動くため、この境界を維持しやすい。

### (4) 医師法・歯科医師法・療養担当規則との関係（論点整理）

- 本ツールは決定木による**事務的な届出可否の自己点検支援**であり、診療行為・診断ではない。
- 「届出代行」（行政書士・社労士の業務領域）や「個別具体の診療報酬請求指導」と受け取られない設計が論点。規約で「本サービスは届出書類の作成代行・提出代行・個別の請求指導を行わない」と明記し、判定は一般化された要件チェックである旨を示すのが安全。社労士法・行政書士法との関係は専門家確認推奨。

### (5) 賠償リスクと備え（一般論）

- 想定リスク＝誤判定による返戻・自主返還・再届出コスト、改定追従漏れ。一般的な備えとして IT事業者賠償責任保険／個人情報漏えい保険の付保が公開事例で見られる。補償範囲・免責金額は約款確認と保険会社・弁護士相談が必要。

| 監修者 | 監修対象 | 粒度 | 依頼時に渡す資料 |

**監修の進め方**：①一次情報（告示/通知）との逐条突合チェックリスト作成 → ②`verify` 項目を優先レビュー → ③改定2年周期での再監修契約（スポット監修＋改定時集中監修の二段構え）。費用感は公開情報が乏しく未確認のため、複数者から相見積もりを取得すること（人手対応）。

> 監修の入力として、本アプリは `verify_flags` と条件単位の `verify` フラグで「要確認」項目を機械的に列挙できる状態になっている。監修者にはこの一覧を渡せば優先順位が明確になる。

## 5. セクション7: 運用・保守（改定への追従）

- **改定周期**：診療報酬改定は2年周期（次回は令和10年度＝2028年6月想定）。ただし中間年でも訂正通知・疑義解釈・事務連絡が随時発出される。令和8年は3月5日告示後、4月2日・5月1日・5月29日に訂正、疑義解釈その1〜その7が発出済み。**「告示確定」だけでなく「訂正・疑義解釈の差分監視」を運用に組み込む必要がある。**
- **監視対象URL**：厚労省「令和8年度診療報酬改定について」（`mhlw.go.jp/stf/newpage_67729.html`、訂正・疑義解釈の集約ページ）、各厚生局の令和8年度届出一覧ページ（更新日表示あり）。RSSは基本的に無いため、ページ更新日・PDFのバイト数/ハッシュ差分監視が現実的。
- **電子申請システム連携**：「保険医療機関等電子申請・届出等システム」は令和8年1月26日時点で324施設基準が電子申請対象（従来113）。GビズID認証対応。**公開API（外部システム連携）は確認できず**、対象一覧はシステム概要PDFから取得する形。アプリからの自動連携は現時点で不可と判断し、「電子申請可否フラグ」表示＋システムへの導線提供に留めるのが現実的。
- **様式PDF URL命名規則による半自動再取得**：近畿・関東信越は `r8-t{様式}.pdf` / `r8-k{様式}.pdf` / `r8-{1or2}-{整理番号}.pdf` の規則性があり、年度プレフィックス（r6→r8）と整理番号を変数化すれば改定時の再取得を半自動化できる。ただし**整理番号は局別・改定別に振り直される**ため、一覧ページのHTMLパースで対応表を毎改定再生成する設計が必要。8局すべてが同一命名規則とは限らない点に注意。

## 6. セクション8: 市場・事業性

- **市場規模**：歯科診療所は全国に多数（e-Stat「医療施設調査」で都道府県別・年次の確定値を取得可能）。レセコンは約9割の歯科医院が導入済み（ITトレンド・2025年9月）。電子レセプト利用率は医療機関・薬局全体で約95%。
- **競合・類似サービス**：
  - レセコン付帯の算定チェック機能（Opt.one、DOC-5、Dentis、MIC WEB SERVICE、POWER5G、Sunny 等、国内40社超）。算定漏れ・返戻防止が主眼で、**「届出可否の事前診断」は主機能ではない**＝本アプリの差別化余地。
  - クラウド型レセコンの価格帯（公開事例）：月額1万円台前半〜4万円程度、初期費用0〜60万円超。
  - 施設基準取得支援は保険医協会・コンサルが研修・サポートを提供（非SaaS）。
- **想定ユーザーと課金モデル**：院長・事務長・歯科衛生士。レセコン市場の価格慣行に照らすと、フリーミアム（基本判定無料＋詳細レポート/様式直リンク/改定追従通知を有料）または低額月額（数千円〜1万円台）が参入しやすい。改定年に需要が集中するため、改定前後のスポット課金や年額プランも検討余地。

## 7. セクション9: 技術・デプロイ運用

- **3省2ガイドライン該当性**：厚労省「医療情報システムの安全管理に関するガイドライン 第6.0版」＋経産省・総務省「医療情報を取り扱う情報システム・サービスの提供事業者における安全管理ガイドライン」の総称。**患者の診療情報（要配慮個人情報）を保存・処理する場合**はベンダ側ガイドラインの対象。本アプリが**患者個票を扱わず医院の体制情報のみを扱う設計**なら直接適用は限定的になり得るが、医院名等の個人情報保護法対応は必要。データ保管は国内リージョン推奨。
- **Next.js + Supabase + Vercel 構成のベストプラクティス（一般）**：
  - 認証：Supabase Auth（医院単位のテナント分離、RLS=Row Level Security でテナント越境防止）。
  - データ保存：最小化（判定は決定木でステートレス化、保存するなら暗号化・国内リージョン）。
  - 通信：TLS必須。APIキー（LLM等）はサーバ側（Route Handler / Server Actions）に限定しクライアント露出禁止、環境変数で管理。
  - ログ/監視：アクセスログ・監査ログの保全、異常検知。
  - LLM（不可理由の解説生成のみ）：コスト/レート制御（キャッシュ・トークン上限）、**プロンプトインジェクション対策**（ユーザー入力をシステムプロンプトから分離、出力は解説のみで判定を上書きしない制約）、判定根拠は決定木出力に固定。
  - バックアップ：ランサムウェア対策としてオブジェクトロック等。

> 現状、`/api/explain` は APIキーをサーバ側 Route Handler に限定し、未設定時はルールベースにフォールバックする実装になっており、上記方針と整合している。永続化・認証・デプロイは未着手。

## 8. 実装で得た知見（設計メモ）

机上の計画ではなく、実装して分かった事実。

- **排他グループ（exclusive_group）は「基準単位」で持つのが正解だった。** 当初は点数項目（line）単位での排他を検討したが、1つの基準内に初診/再診など複数の line があり、line 単位で排他すると同一基準内の初診と再診まで打ち消し合ってしまう。結局「同月にどちらか一方しか算定しない基準どうし（外感染1〜4／外安全1・2／歯初診・病初診／歯援診・歯援病）」を `revenue_sim.exclusive_group` として基準単位で持ち、合算時に**グループ内で収益最大の基準だけを採用、他は除外行として残す**形に落ち着いた。
- **「月1回」は総回数のキャップではない。** 「月1回」は患者1人あたりの制約であり、医院全体の月間延べ算定回数は患者数だけ増える。したがって `once_per_month` は**回数を制限せず表示で明示するだけ**にした（回数欄が延べ回数であることを伝える役割）。総回数を1にキャップする実装は誤り。
- **みなし期間は文字列でなく期限日（transitional_deadline）で持つと価値が出る。** `transitional` の文章とは別に `YYYY-MM-DD` を持たせ、システム日付と比較して「あと○日」を出すことで、改定直後の「いつまでに体制を整えればよいか」が一目で分かる。期限切れは「みなし終了済み」に自動で切り替わる。
- **整理番号・様式PDFは単一値で持つと不正確になる。** 局・年度で振り直されるため、`code_number_bureau` で出所を明示し、UIで「管轄厚生局で要確認」を常時表示する設計にした。本格運用では「厚生局×年度」の多次元で持つべき（現状は確認した1〜2局の値）。

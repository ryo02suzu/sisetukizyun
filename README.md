# 歯科 施設基準 届出可否 診断アプリ（令和8年度改定対応）

歯科診療所が届出可能な施設基準について、**届出可否を決定木で判定**し、**収益を係数調整型で試算**するWebアプリです。令和8年度（2026年6月施行）改定に対応しています。

> 構造化調査・実装パッケージ（厚労省告示第69〜71号 / 通知 保医発0305第6〜8号 / 近畿厚生局 令和8年度届出様式）に基づきます。

## 主な機能

- **問診**：設備・人員・体制・実績・研修を、カテゴリ別の折りたたみアコーディオン＋回答進捗バーで入力。セクション単位の一括回答にも対応。回答は端末に自動保存（localStorage）。
- **判定結果**：15件の施設基準を「届出可能／要確認／届出不可」で判定。サマリーのクリックで絞り込み。各カードで不足要件・要確認の論点・前提基準を表示し、「点数・様式・出典」の詳細を展開できる。
- **解説**：不可理由・要確認の論点を自然言語で言い換え（判定はしない）。
- **収益試算**：対象基準をチェックで選び、月間算定回数を入力して月額・年額を試算。
- **印刷 / PDF**：判定結果・試算をチェックリストとして出力（ナビ等は印刷時に非表示）。
- **モバイル対応**：スマートフォン幅にレスポンシブ対応。

## 設計方針

このアプリは「間違えてはいけない確定情報（点数・要件）」と「言葉での説明」を明確に分離しています。

| レイヤー | 役割 | 実装 | LLMを使うか |
| --- | --- | --- | --- |
| データ | 施設基準を統一スキーマで構造化 | `data/standards.ts` | ✕ |
| 判定エンジン | 要件を決定木で評価し届出可否を判定 | `lib/engine.ts`（純粋関数） | ✕ |
| 収益試算 | 点数 × 月間算定回数 × 10円 | `lib/revenue.ts` | ✕ |
| 解説生成 | 不可理由・要確認の論点を日本語で言い換え | `lib/explain.ts` + `/api/explain` | ○（解説のみ） |
| UI | 問診 → 判定結果 → 収益試算 | `app/page.tsx` ほか | ✕ |

- **判定はLLMにさせません。** ルールエンジンが決定論的に判定し、LLMは結果の言い換え（解説）にのみ使います。これにより返還リスクを避けます。
- 解釈が割れる項目（「常勤」の定義 / 研修の有効性 / 経過措置 / 未確定の整理番号など）は `verify` フラグで **「要確認」** に倒しています。
- 点数・様式番号は確定情報を優先し、不確実なものは各基準の `verify_flags` に明示しています。

### 環境変数（すべて任意）

- `ANTHROPIC_API_KEY` … 「解説を表示」のAI解説に使用。**判定には一切使いません。** 未設定時はルールベースの解説文にフォールバックします。
- `ANTHROPIC_MODEL` … 既定 `claude-sonnet-4-6`。
- `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` … 将来の永続化用（後述）。未設定時はローカル JSON（`data/standards.ts`）をデータ源として動作します。

## データモデル（統一スキーマ）

`lib/types.ts` の `DentalFacilityStandard` が1件分の施設基準です。要件は5カテゴリ（`equipment` 設備 / `staff` 人員 / `system` 体制 / `performance` 実績 / `training` 研修）に分かれ、各 `Condition`（決定木ノード）は次の `type` を取ります。

- `boolean` … はい/いいえ
- `number_min` / `threshold` … 数値が閾値以上
- `composite_or` … 子条件のいずれか1つ以上（OR）
- `composite_and` … 子条件のすべて（AND）

`prerequisites` は前提となる他基準の `id`（例：歯科外来診療感染対策加算は `ha_shoshin` が前提）。判定は前提解決のため反復評価して収束させます。

## Supabase への移行（任意）

現在はローカル JSON をデータ源にしています。パッケージの設計に沿って Supabase（PostgreSQL）へ移す場合：

- `facility_standards` … 施設基準を JSONB で保持（`last_updated` / `source_version` を列に持つ）
- `user_inputs` … 問診回答を key-value で保存
- `diagnosis_results` … 判定結果（可否・要確認・不可）を保存

`data/standards.ts` のオブジェクト構造をそのまま `facility_standards` の JSONB にマッピングできます。

## データの検証状況（令和8年度・一次情報照合）

`data/standards.ts` の各施設基準は、地方厚生局の令和8年度届出様式一覧（近畿・関東信越・東海北陸・九州）と厚労省・しろぼんねっと令和8年点数表を中心に照合しています。

- **整理番号・届出様式・本体点数**は、原則として厚生局2機関以上＋点数表でクロスチェックした確定値を採用（例：歯初診 1-16／様式2の6、外感染1〜4 1-20〜1-23、歯医DX1 1-8／様式1の6、口管強 2-89／様式17の2、ベースアップⅠ 2-611／様式95 等）。
- **歯科初診料（注1）** は「加算」ではなく**減算方式**（届出あり272点／なし245点、再診59/45点）。収益試算は届出による差額（初診27点・再診14点）で計算します。
- **追加照合（厚労省個別改定項目・疑義解釈・しろぼんねっと等）で確定**：マイナ保険証利用率30%以上、CAD/CAMインレー770点・CAD/CAM冠1,200点、歯科技工士連携加算 60/80点（2025年4月以降）、光学印象歯科技工士連携加算50点（対面単一）、咀嚼能力検査（D011-2）・咬合圧検査（D011-3）の施設基準廃止、GTR整理番号2-484、歯科治療時医療管理料45点据え置き、医療機器安全管理料（歯科）B018 1,100点、ベースアップ評価料の対象職員拡大・（Ⅱ）5割未満要件。施行は令和8年6月1日（薬価のみ4月1日）。
- **最終確定（6項目の追加照合）**：歯医DXの歯科の規定位置＝**初診料A000注15・再診料A002注12**（訂正）、医療機器安全管理料（歯科）**整理番号2-86・様式15**、歯援診1=**2-92**/2=**2-93**・様式18、地域医療連携体制加算**300点**を確定。歯援診の実績要件は令和8年で「過去1年18回」から「直近1か月・4ルート選択制」へ変わったため、判定ロジックも選択制（OR条件）へ作り替え済み。
- **要確認のまま残る項目**（一次原本に到達できず）：GTR・口腔粘膜処置・レーザー機器加算の令和8点数本文（据え置きの公算）、地域医療連携体制加算の整理番号・様式、歯援診の訪問実績回数の告示逐語、整理番号の他厚生局での一致、各基準の令和8固有の経過措置の逐語。各 standard の `verify_flags` と条件の `verify` で明示しています。
- 各 standard の `sources` に照合に用いた一次情報のURLを記載しています。

> 注：厚労省の告示・通知PDFは日本語フォントの都合で機械抽出できない箇所があり、その場合はしろぼんねっと（告示の転記）や厚生局の様式PDF実物を一次的根拠としています。最終的な届出にあたっては告示原本・厚生局の最新様式での確認を推奨します。

## 免責

判定・試算は入力情報に基づく目安です。実際の届出にあたっては、**必ず最新の告示・通知および地方厚生（支）局の届出様式で最終確認**してください。点数・様式番号・経過措置・「常勤」の定義など解釈が分かれる項目は「要確認」として表示しています。
